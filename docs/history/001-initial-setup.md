## 1. 프로젝트 개요

### 프로젝트명
- **Build-High (빌드-하이)**  
  - 이 히스토리 문서는 “Build-High”라는 문제의식에서 출발한 프로젝트의 탄생 기록이다.
  - 다만, **현재 리포지토리의 핵심 설계 문서(`docs/PRD.md`, `docs/FLOW.md`)는 DY Carbon(산업용 카본·그래파이트 B2B 웹 플랫폼) MVP 흐름을 기준으로 작성**되어 있으며, 본 문서에서도 그 근거를 그대로 따른다.

### 목적
- **대학생 / 취준생을 위한 사이드 프로젝트 팀원 모집 및 커뮤니티 플랫폼**

### 문제 정의
- 기존 팀원 모집 글은 정보가 흩어져 있고,  
  **직군·기술·기간·목표가 한눈에 파악되지 않아** 매칭의 비용이 높다.

### 핵심 가치
- “**5가지 핵심 데이터 칩(직군, 기술 스택, 프로젝트 기간, 목표, 진행 상태)**”를 통해  
  팀 모집 정보를 **명확하고 구조적으로 전달**한다.

### Phase 1 설계 철학(완주 중심)과의 연결
`docs/PRD.md`가 강조하는 Phase 1의 철학은 “완성도 높은 큰 서비스”가 아니라,  
**명확한 사용자 흐름을 끝까지 구현한 MVP를 완주하는 것**이다.

- 그 철학을 Build-High에 적용하면:
  - 무한한 커뮤니티 기능을 먼저 늘리기보다,
  - 사용자가 “정보를 이해하고(탐색) → 행동하고(작성) → 결과를 얻는(저장/확인)” 흐름을
    **끊김 없이 경험**하게 만드는 것이 우선순위가 된다.
- 현재 리포지토리의 PRD/FLOW는 이 “완주 가능한 MVP” 철학을,
  **제품 탐색 → 문의 전환 → AI 응답 저장 → 마이페이지 확인**이라는 단일 여정으로 구체화한다.

## 2. 기술 스택 선정 이유 (Tech Stack)

아래 기술 선택은 “유행”이 아니라, `docs/PRD.md`가 정의한 **Phase 1 MVP 완주**와  
`docs/FLOW.md`가 고정한 **사용자 여정/시스템 흐름을 안정적으로 구현**하기 위한 합리적 결정이었다.

### Framework — Next.js 15 (App Router)
- **왜 Next.js인가**
  - `docs/FLOW.md`의 페이지 구조(메인 → 로그인 → 리스트/상세 → 작성 → 마이페이지)는
    **파일 기반 라우팅**과 **라우트 그룹**이 자연스럽게 맞는다.
  - Server Components 중심으로 “읽기(Products)”와 “쓰기(Inquiry/AI)”를 분리해,
    **초기 번들 부담을 줄이면서도 확장 가능한 구조**를 유지할 수 있다.
- **결과적으로 얻는 것**
  - SEO/공유에 유리한 기본 구조
  - 사용자 흐름이 늘어나도 흔들리지 않는 라우팅/레이아웃 뼈대

### Styling — Tailwind CSS + Lucide React
- **왜 Tailwind인가**
  - MVP는 화면 수가 빠르게 늘어날 수밖에 없고(Flow 기반 페이지 확장),
    스타일을 “규칙”으로 고정해 **일관된 속도로 UI를 완성**해야 한다.
  - v0/shadcn 계열 결과물을 이식할 때 비용이 낮다(유틸 중심).
- **왜 Lucide React인가**
  - 아이콘 품질/일관성을 확보하면서도, 디자인 시스템을 과도하게 키우지 않고
    **B2B 톤의 신뢰감 있는 UI**를 빠르게 구성할 수 있다.

### Backend / DB — Supabase
- **왜 Supabase인가**
  - `docs/PRD.md` Phase 1 요구사항(인증 + DB CRUD + 최소한의 정책)을
    **하나의 서버리스 묶음(Auth + Postgres + RLS)** 으로 빠르게 완성할 수 있다.
  - `docs/FLOW.md`의 흐름처럼
    - Google OAuth 로그인 → user upsert
    - 제품 read-only 조회
    - 문의 insert/update/list/delete
    를 단일 백엔드 경험으로 연결하기 좋다.
- **결과적으로 얻는 것**
  - “최소 기능 완주”에 필요한 인프라 결정을 줄이고,
    제품/흐름 자체에 집중할 수 있는 개발 리듬

### Package Manager — pnpm
- **왜 pnpm인가**
  - 의존성 설치 속도와 디스크 효율이 좋고,
    Phase 2 확장(패키지/도구 추가)에도 무리 없는 선택이다.
  - v0/shadcn 이식 과정에서 여러 UI 의존성이 추가되더라도,
    설치/관리 비용을 낮춰준다.

### AI Tooling — Cursor
- **왜 Cursor인가**
  - 이 프로젝트는 문서가 소스 오브 트루스이며(`docs/PRD.md`, `docs/FLOW.md`),
    Phase 1 MVP는 “끝까지 이어지는 흐름”을 빠르게 완성해야 한다.
  - Cursor는 PRD/FLOW 기반으로 문서와 코드를 함께 진화시키는
    **Vibe Coding(가속) 전략**에 적합했다.
- **협업 원칙**
  - AI는 “속도”를 돕지만, **결정은 사람이 한다**(규칙/범위/흐름은 고정).

## 3. 초기 환경 구축 내역

### Boilerplate
- **Next.js App Router 기반 기본 구조 세팅**
  - 라우트 그룹(인증/대시보드)과 전역 레이아웃을 전제로 시작
- **디렉토리 구조 선확립**
  - `app/` (라우팅/서버 액션/핸들러)
  - `components/` (UI/도메인/레이아웃 컴포넌트)
  - `lib/` (Supabase 클라이언트, 유틸, AI 클라이언트 등)
  - `types/` (도메인/DB/API 타입의 기준점)
- **확장 가능한 구조 우선**
  - Phase 2는 Phase 1의 구조를 기반으로 확장한다는 원칙(`docs/PRD.md`)에 맞춰,
    “당장 쓰는 것만”이 아니라 **나중에 커질 때 무너지지 않는 뼈대**를 먼저 만든다.

### Design Assets
- **v0.dev를 통해 초기 UI 시안 생성**
  - 빠르게 형태를 만들고, 구현 가능한 컴포넌트 자산으로 전환
- **직접 수정·이식 가능한 자재로 활용**
  - 필요 이상으로 추상화하지 않고,
    화면/흐름 완주를 위해 실용적으로 가져온다.

### Project Rules
- **`.cursor/rules/*.mdc` 기반 규칙 정의**
  - PRD 우선, FLOW 우선: 코드가 문서를 앞서가지 않도록 고정
  - UI 톤/일관성: 과한 커스텀 대신 검증된 패턴 중심
  - 품질 기준: 입력 검증/에러 처리/상태 UI(로딩·실패·빈 상태)를 기본값으로
- **사람-중심 협업 원칙**
  - AI는 보조 수단이며,
  - 범위/흐름/제품 결정은 사람이 내린다(문서에 기록하고 코드로 구현).

## 4. 주요 문서화 현황

### `docs/PRD.md`
- **역할**: Build-High(문제의식)에서 출발해,
  실제 구현 가능한 Phase 1 범위를 “완주 가능한 MVP”로 고정하는 기준선.
- **포함**:
  - 서비스 목적/문제 정의
  - Phase 1 / Phase 2 범위
  - 핵심 시나리오(로그인 → 탐색 → 작성 → 저장/확인)와 데이터 모델

### `docs/FLOW.md`
- **역할**: “사용자가 실제로 어떻게 움직이는가”를 기준으로,
  페이지 구조/인증/데이터 흐름을 **한 장으로 설명하는 설계도**.
- **포함**:
  - 사용자 여정(페이지 이동)과 시스템 상호작용(Sequence)
  - 제품 조회(read-only)와 문의/AI 저장(write)의 책임 분리

## Appendix. FLOW/PRD에서 고정한 ‘첫 완주’의 형태(요약)

`docs/FLOW.md`는 Phase 1 MVP를 다음의 단일 흐름으로 고정한다.

- **페이지 흐름(요약)**: `/` → `/login` → `/products` → `/products/:id` → `/inquiry?product=:id` → `/my-page`
- **로직 흐름(요약)**:
  - 로그인 후 사용자 upsert
  - 제품은 read-only 조회
  - 문의는 먼저 INSERT(ai_response=null) → AI 생성 → 같은 문의 UPDATE로 ai_response 저장 → 마이페이지에서 목록/삭제

이 “끝까지 이어지는 완주”가, 이후 어떤 방향(Build-High든 다른 도메인이든)으로 확장하더라도  
프로젝트의 기본 리듬과 품질 기준을 지탱하는 출발점이 된다.
